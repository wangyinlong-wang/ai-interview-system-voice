## 1. 引言

### 1.1 增补背景

现有系统的面试题完全由 AI 根据用户简历实时生成，虽然具备一定的个性化能力，但缺乏结构化、经过验证的面试题库作为支撑。这导致以下问题：

- **题目质量波动大**：纯 AI 生成的问题有时偏离实际面试场景，缺乏行业针对性
- **缺少标准答案参考**：AI 在评估用户回答时，缺乏权威的参考答案进行对比
- **知识更新滞后**：无法及时获取各公司最新的面试趋势和热门题型
- **内容沉淀不足**：系统缺乏可积累、可复用的结构化面试知识资产

### 1.2 增补目标

通过引入定时爬虫 + 面试知识库模块，实现：

1. **每日自动采集**：从多个权威渠道自动爬取最新面试题，保持题库鲜活
2. **智能分类存储**：按岗位、难度、题型自动归类，构建结构化知识库
3. **AI 增强提问**：AI 面试官参考题库中的真实面试题，生成更贴近实际的问题
4. **评估有据可依**：评估时参考知识库中的标准答案和解析，提升评估准确性
5. **人工审核机制**：提供管理后台，支持运营人员对题库内容进行审核和编辑

### 1.3 影响范围

| 影响模块 | 影响说明 |
|----------|----------|
| AI 面试引擎 | 新增知识库检索能力，生成问题时参考真实题库 |
| 评估模块 | 评估时对比知识库中的标准答案 |
| 管理后台 | 新增题库管理、采集日志、来源配置等页面 |
| 用户端 | 新增面试题库浏览页面（/question-bank） |
| 数据库 | 新增 questions、question_categories、crawl_sources、crawl_logs 等表 |
| 服务端 | 新增定时调度任务（爬虫引擎）、数据清洗管道 |

## 2. 面试题库采集模块（章节 4.10）

### 2.1 模块概述

面试题库采集模块是系统的自动化内容获取引擎，负责从多个互联网来源定时爬取面试相关数据，经过清洗、分类、去重后存入知识库，为 AI 面试引擎提供权威、新鲜、结构化的题目素材。

**核心能力**：
- 支持配置多个爬取来源（URL / API），每日自动执行
- 爬取后自动按岗位类型、题目类型、难度级别进行智能分类
- 基于内容相似度算法自动去重，确保题库质量
- 完整的采集日志记录，便于监控和排查
- 提供管理后台，支持人工审核、编辑、删除题目

```
+------------------------------------------------------------------+
|                    面试题库采集模块架构                             |
+-------------+----------------+----------------+-------------------+
|   调度层     |    采集层       |    处理层       |     存储层         |
|  - 定时任务  |  - 多源适配器   |  - 数据清洗    |  - 题目主表        |
|  - 手动触发  |  - 请求管理    |  - 智能分类    |  - 分类标签表      |
|  - 频率配置  |  - 反爬策略    |  - 相似去重    |  - 采集日志表      |
|  - 优先级   |  - 并发控制    |  - 质量评分    |  - 来源配置表      |
+-------------+----------------+----------------+-------------------+
```

### 2.2 功能需求清单

#### 2.2.1 多源采集（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 来源配置 | 支持在管理后台配置多个爬取来源，包括来源名称、URL、请求方式（GET/POST）、请求头、采集规则（CSS选择器/XPath/正则）、采集频率 |
| 来源启用/禁用 | 支持对每个来源独立启用或禁用，不影响其他来源的采集 |
| 字段映射 | 支持配置来源字段到系统标准字段的映射（如将来源的"title"映射为系统的"question_content"） |
| 自动执行 | 系统按照配置的频率自动触发采集任务，默认每天凌晨 2:00 执行 |
| 手动触发 | 管理员可在后台手动触发单次采集，便于测试和紧急补采 |
| 失败重试 | 单个来源采集失败时自动重试 3 次，仍失败则记录错误并继续下一个来源 |

#### 2.2.2 智能分类（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 岗位分类 | 自动识别题目所属岗位类型，至少覆盖：前端开发、后端开发、算法工程师、移动端开发、测试开发、运维/DevOps、产品经理、UI/UX设计、运营、数据分析师 |
| 题型分类 | 自动识别题目类型：技术八股、算法编程、场景设计、项目深挖、 behavioral（行为面试）、案例分析、开放讨论 |
| 难度标注 | 基于题目内容特征自动标注难度级别（初级/中级/高级），支持后续人工校正 |
| 多维标签 | 支持自动或人工打标签，标签维度包括：技术栈（React/Vue/Java/Go 等）、知识点（HTTP/操作系统/数据库等）、公司类型（大厂/外企/创业公司等） |

#### 2.2.3 去重处理（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 相似度计算 | 使用文本相似度算法（SimHash / MinHash / 余弦相似度）计算题目间相似度 |
| 自动去重 | 相似度超过设定阈值（默认 85%）的题目自动标记为重复，仅保留最早入库的版本 |
| 重复记录 | 重复题目不物理删除，而是记录关联关系，便于追溯 |
| 去重统计 | 每次采集完成后统计去重率，生成去重报告 |

#### 2.2.4 数据清洗（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 广告过滤 | 使用规则和 NLP 模型识别并过滤广告内容、推广信息 |
| 格式标准化 | 统一题目格式，去除多余空格、特殊字符、HTML 标签残留 |
| 长度过滤 | 过滤过长（>2000字）或过短（<10字）的无效内容 |
| 敏感信息过滤 | 使用关键词库过滤涉政、涉黄、暴恐等敏感内容 |
| 质量评分 | 对清洗后的内容进行质量评分（0-100分），低于阈值（默认 60 分）的题目进入待审核状态 |

#### 2.2.5 采集调度（P1）

| 功能点 | 详细描述 |
|--------|----------|
| 采集频率配置 | 支持按来源设置采集频率：每小时/每天/每周/自定义 Cron 表达式 |
| 并发控制 | 支持配置同时采集的来源数量，避免对目标网站造成过大压力 |
| 采集时段限制 | 支持设置采集时段（如仅允许 01:00-06:00），避免高峰期对目标网站造成干扰 |
| 优先级队列 | 支持为不同来源设置采集优先级，核心来源优先执行 |

#### 2.2.6 采集日志（P1）

| 功能点 | 详细描述 |
|--------|----------|
| 执行记录 | 每次采集任务生成一条执行记录，包含：任务ID、开始时间、结束时间、执行状态 |
| 来源明细 | 记录每个来源的采集明细：请求次数、成功数、失败数、新增数、重复数、错误信息 |
| 日志查询 | 支持按时间范围、来源、状态筛选查询采集日志 |
| 日志保留 | 采集日志保留 90 天，超期自动归档 |

#### 2.2.7 人工审核（P1）

| 功能点 | 详细描述 |
|--------|----------|
| 审核列表 | 展示待审核题目列表，按质量评分和采集时间排序 |
| 批量操作 | 支持批量通过、批量拒绝、批量删除 |
| 编辑功能 | 支持编辑题目的题干、参考答案、分类、标签、难度 |
| 审核记录 | 记录审核操作人、审核时间、审核结果、备注 |
| 审核统计 | 展示审核效率统计：待审核数、今日审核数、审核通过率 |

#### 2.2.8 热门排序与收藏（P2）

| 功能点 | 详细描述 |
|--------|----------|
| 引用计数 | 记录每道题目被 AI 面试引用提问的次数 |
| 收藏功能 | 用户可收藏题目（后续版本），收藏数作为热门度参考 |
| 热门排序 | 题库浏览时支持按引用次数、收藏数、入库时间排序 |

## 3. 面试知识库模块（章节 4.11）

### 3.1 模块概述

面试知识库是系统的结构化知识中枢，存储经过清洗、分类、审核的面试题目及其参考答案、解析。知识库为 AI 面试引擎提供权威的内容素材，使 AI 面试官能够基于真实面试题生成更专业、更贴近实际的问题，并在评估阶段提供参考答案进行比对。

```
+-------------------------------------------------------------+
|                      面试知识库架构                           |
+-------------+-------------------+---------------------------+
|   存储层     |    检索层          |      应用层                |
|  - 题目主表  |  - 全文检索       |  - AI 面试引擎（生成问题）  |
|  - 分类表   |  - 向量检索       |  - 评估引擎（对比答案）     |
|  - 标签表   |  - 混合排序       |  - 题库管理后台            |
|  - 答案表   |  - 语义匹配       |  - 用户端题库浏览          |
|  - 引用记录 |  - 缓存加速       |  - 统计分析                |
+-------------+-------------------+---------------------------+
```

### 3.2 功能需求清单

#### 3.2.1 题目存储（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 题目内容 | 结构化存储题干、参考答案、详细解析 |
| 元信息 | 存储岗位分类、题型分类、难度级别、多维标签 |
| 来源追溯 | 记录来源网站、原始 URL、采集时间、入库时间 |
| 状态管理 | 题目状态：待审核 / 已通过 / 已拒绝 / 已下架 |
| 版本控制 | 记录编辑历史，支持查看和回滚 |

#### 3.2.2 知识库检索（P0）

| 功能点 | 详细描述 |
|--------|----------|
| 语义检索 | AI 面试引擎根据岗位、难度、用户简历关键词，从知识库检索最相关的 N 道题目 |
| 全文检索 | 管理后台支持按关键词全文检索题目内容 |
| 组合筛选 | 支持按岗位 + 难度 + 题型 + 标签 + 时间范围组合筛选 |
| 缓存加速 | 热点检索结果缓存，降低数据库压力和响应延迟 |
| 检索日志 | 记录检索 query、结果数量、耗时，用于优化检索效果 |

#### 3.2.3 题目引用追踪（P1）

| 功能点 | 详细描述 |
|--------|----------|
| 引用记录 | AI 面试使用某道题目生成问题时，记录引用关系 |
| 引用统计 | 统计每道题目的被引用次数，用于热门排序 |
| 效果追踪 | 追踪基于题库生成的问题 vs 纯 AI 生成的问题的用户满意度差异 |

#### 3.2.4 知识库统计（P1）

| 功能点 | 详细描述 |
|--------|----------|
| 总量统计 | 总题量、各岗位题量、各难度题量 |
| 增量统计 | 今日新增、本周新增、本月新增 |
| 质量统计 | 各来源去重率、平均质量分、审核通过率 |
| 使用统计 | 被引用次数分布、热点岗位题型 |

#### 3.2.5 答案补充（P2）

| 功能点 | 详细描述 |
|--------|----------|
| 标准答案补充 | 管理员可为题目补充或编辑标准答案和详细解析 |
| 多答案版本 | 支持存储多个版本的参考答案（如初级回答、高级回答） |
| 答案引用来源 | 记录答案的来源（管理员编写 / 采集自动提取 / 用户贡献） |

```
|    （问题生成）             |
+---------------------------+
        |
        v
+---------------------------+
| 5. 记录引用关系            |  记录本次面试引用了哪些知识库题目
|    （引用追踪）             |
+---------------------------+
```

## 4. 数据库模型补充

### 4.1 新增数据表设计

#### 4.1.1 题目主表（questions）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | BIGINT PK | 主键，自增 |
| content | TEXT | 题干内容 |
| answer | TEXT | 参考答案 |
| analysis | TEXT | 详细解析 |

## 6. AI 集成更新：面试引擎与知识库集成

### 6.1 集成架构

```
+-------------------------------------------------------------+
|                     AI 面试引擎（更新）                       |
+-------------+----------------------+----------------------+
|  原有能力    |    新增能力（知识库集成）|    输出              |
|  - 简历解析  |  - 知识库语义检索      |  - 面试问题          |
|  - 个性化   |  - 题目上下文注入      |  - 追问问题          |
|  - 评估模型 |  - 标准答案对比        |  - 评估反馈          |
|  - 追问逻辑 |  - 引用追踪记录        |  - 引用来源标注       |
+-------------+----------------------+----------------------+
```

### 6.2 集成方式

#### 6.2.1 面试问题生成阶段

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 检索知识库 | 用户创建面试时，根据岗位类型 + 难度级别 + 简历关键词，从知识库检索 Top-10 最相关题目 |
| 2 | 构建上下文 | 将检索到的题目格式化，注入 System Prompt 的知识库参考区域 |
| 3 | 生成问题 | AI 参考这些真实面试题的风格和考察点，结合用户简历个性化生成问题 |
| 4 | 记录引用 | 记录本次问题生成引用了哪些知识库题目 |

**注入 System Prompt 的知识库上下文示例**：

```
【知识库参考题目】
以下是从知识库中检索到的与用户岗位和技能相关的真实面试题，请在生成问题时参考这些题目的风格和考察点：

1. [后端开发-中级] Redis 缓存穿透、缓存击穿、缓存雪崩的区别和解决方案？
2. [后端开发-中级] MySQL InnoDB 的索引数据结构是什么？为什么选择 B+ 树？
3. [后端开发-中级] 如何设计一个高可用的分布式锁？

请基于以上参考题目，结合用户简历内容，生成 1 道个性化的面试问题。
要求：问题的考察点应与参考题目类似，但需结合用户的项目经历进行个性化定制。
```

#### 6.2.2 面试评估阶段

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 检索标准答案 | 评估用户回答时，从知识库中获取当前题目对应的标准答案和解析 |
| 2 | 对比评估 | AI 将用户回答与标准答案进行对比，从准确性、完整性、深度等维度评估 |
| 3 | 生成点评 | 基于对比结果生成更精准的逐题点评 |
| 4 | 改进建议 | 参考标准答案中的要点，给出针对性的改进建议 |

### 6.3 集成效果指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 知识库引用覆盖率 | ≥ 60% | AI 面试中引用知识库题目的面试占比 |
| 用户满意度提升 | ≥ 10% | 引用知识库的题目 vs 纯 AI 生成题目的用户评分差异 |
| 评估准确率提升 | ≥ 15% | 有标准答案参考的评估 vs 无参考评估的准确度差异（通过专家抽样评估） |

| 单来源采集耗时 | ≤ 5 分钟 | 单个来源的完整采集时间 |

| 请求间隔 | ≥ 1 秒 | 对单个目标网站的连续请求间隔 |

| 日志保留 | 采集日志保留 90 天 | 超期自动归档或清理 |
| 缓存命中率 | ≥ 60% | 热点检索结果的缓存命中率 |

### 7.3 稳定性与容错

| 需求项 | 要求 |
|--------|------|
| 爬虫容错 | 单个来源采集失败不影响其他来源，失败来源记录错误日志并发送告警 |
| 服务降级 | 知识库服务不可用时，AI 面试引擎自动降级为纯 AI 生成模式，不影响核心功能 |
| 数据一致性 | 采集、清洗、分类、去重、入库全流程事务保证，避免数据丢失或不一致 |
| 限流保护 | 对目标网站的请求严格限流，避免因爬取导致目标网站异常 |
| 监控告警 | 采集失败、去重率异常、知识库检索延迟超标等场景自动触发告警 |

### 7.4 安全与合规

| 需求项 | 要求 |
|--------|------|
| robots.txt 遵守 | 爬取前检查目标网站的 robots.txt，遵守其中的爬取限制 |
| 请求频率控制 | 对单个来源的请求频率严格控制在合理范围，避免对目标网站造成压力 |
| User-Agent 声明 | 使用可识别的 User-Agent，包含系统名称和联系方式 |
| 数据用途限定 | 爬取的数据仅用于系统内部学习和为用户提供面试服务，不对外传播或商用 |
| 版权声明 | 题库展示页面标注数据来源，尊重原作者版权 |
| 隐私保护 | 不爬取、不存储任何个人隐私信息（如真实姓名、手机号、身份证号等） |
| 反爬应对 | 遇到验证码、IP 封禁等反爬措施时自动暂停，不强行突破 |
| 合法来源优先 | 优先使用官方 API、RSS 订阅、开放数据等合法渠道获取内容 |

## 9. 风险与应对（增补）

| 风险类别 | 风险描述 | 影响程度 | 应对措施 |
|----------|----------|----------|----------|
| 合规风险 | 爬取目标网站数据可能违反其服务条款或 robots.txt | 高 | 严格遵守 robots.txt；优先使用官方 API；设置合理请求频率；User-Agent 中注明身份和联系方式；发现禁止爬取时立即停止 |
| 合规风险 | 爬取内容可能涉及版权争议 | 中 | 仅用于系统内部学习和用户面试服务，不对外传播；展示页面标注数据来源；收到投诉立即下架相关内容 |
| 技术风险 | 目标网站反爬升级导致采集失败 | 中 | 设计可插拔的适配器架构，单个来源失败不影响整体；监控采集成功率，异常时自动告警并人工介入 |
| 技术风险 | 爬取数据质量不稳定，大量无效数据 | 中 | 完善数据清洗管道和质量评分模型；低质量数据进入待审队列不直接入库；持续优化清洗规则 |
| 产品风险 | 知识库题目与 AI 生成问题融合不自然 | 中 | A/B 测试对比知识库辅助 vs 纯 AI 生成的效果；根据用户反馈持续优化 Prompt 和检索策略 |
| 运营风险 | 题库内容审核工作量大 | 中 | 提高自动审核通过率（质量评分模型优化）；分批处理，优先审核核心岗位题目；运营人员逐步积累审核效率 |

| 准确率 | 检索系统中，返回结果中真正相关的比例 |
